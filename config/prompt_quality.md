## Режим: QUALITY (Caution over Speed)

### Приоритеты (строго по порядку):

1. **Security** — поиск и устранение уязвимостей (OWASP Top 10, инъекции, auth bypass)
2. **Bug fixes** — исправление реальных багов, edge cases, race conditions
3. **Test coverage** — добавление тестов для критических путей без покрытия
4. **Code quality** — рефакторинг опасного/хрупкого кода, устранение code smells
5. **Stability** — обработка ошибок, валидация, graceful degradation

### Правила выполнения:

- **Каждое изменение должно иметь тест** — без исключений для кода
- **Security-first mindset** — каждый input/output/check на предмет уязвимостей
- **Не ломай существующее** — все текущие тесты должны проходить. Если тесты падают даже не из-за твоих изменений — почини их. Запрещено игнорировать упавшие тесты с обоснованием "это не я сломал".
- **Проверяй边界 условия** — empty input, null, overflow, concurrency
- **Один эксперимент = одно конкретное исправление/улучшение качества**
- **Document trade-offs** — если оставляешь TODO, объясни риск

## Security Checklist

Перед каждым изменением проверь:

- [ ] **Input validation** — все пользовательские входы валидируются?
- [ ] **Injection** — нет SQL/command/XSS/SSRF инъекций?
- [ ] **Auth/Authz** — доступ к защищённым ресурсам проверяется?
- [ ] **Data exposure** — нет утечки чувствительных данных в логах/responses?
- [ ] **Dependencies** — нет использования устаревших/уязвимых паттернов?
- [ ] **Error handling** — ошибки не раскрывают внутреннее состояние системы?
- [ ] **Path traversal** — нет обхода файловой системы через `../`?

## Complexity Budget

**Правило**: Файлы должны быть разделены по функциональной ответственности или бизнес-значимости. Разделяй — это лучше, чем один монолитный файл на тысячи строк.

Прежде чем создать НОВЫЙ файл:
1. **Есть ли у нового файла чёткая единственная ответственность?** (один модуль = одна задача)
2. **Новый файл логически самостоятелен?** (его можно переиспользовать или читать отдельно)
3. **Улучшает ли это читаемость?** Разбивай, когда это имеет смысл

Прежде чем добавить в СУЩЕСТВУЮЩИЙ файл:
1. **Связан ли код с текущей ответственностью файла?** Если нет — создай отдельный
2. **Не превращается ли файл в монолит?** Если да — разбивай
3. **Улучшит ли это читаемость?** Один огромный файл хуже, чем несколько по смыслу

**Анти-паттерн:** "Всё в одном файле, потому что так проще найти" — в итоге ничего не найти.
**Паттерн:** Разделяй по модулям/компонентам, соединяй через импорты/включения.

**Quality-specific:** При рефакторинге с разбиением файлов — сначала напиши регрессионный тест на поведение, затем разделяй.

### 1. Find Issues

Прочитайте память проекта и поищите проблемы:

**Приоритетные направления (чередуй):**

3. **Test gaps** — найди критический код без тестов, напиши
4. **Error handling** — найди silent failures, добавь обработку
5. **Code smells** — дублирование, god functions, magic numbers

### 3. Implement & Test

**Порядок (строгий):**
1. Напиши тест, воспроизводящий проблему (RED)
2. Внеси исправление (GREEN)
3. Убедись что тест проходит
4. Запусти ВСЕ существующие тесты — ни один не должен упасть. Если упал тест, не связанный с твоим изменением — всё равно почини его. Не игнорируй.
5. Проверь edge cases

**Что МОЖНО менять:**
- Любые файлы в проекте
- Добавлять тесты в любые модули
- Улучшать error handling и validation

**Что ЗАПРЕЩЕНО:**
- `git push` — без явного запроса
- `git reset --hard` — потеря изменений
- Удаление `.autoresearch/` или `.claude/memory/`
- Изменения ВНЕ проекта
- Менять функциональное поведение без теста
- Удалять код без регрессионного теста

### Git Commit (ОБЯЗАТЕЛЬНО после каждого эксперимента)

После завершения работы и перед отчётом — зафиксируй изменения в git:

```bash
git add -A
git commit -m "exp #{iteration}: <краткое описание что изменено>"
```

**Правила:**
- Коммит делается ВСЕГДА, даже если решение DISCARD — откат тоже фиксируется в истории
- Сообщение коммита: `exp #N: краткое описание` (N = номер эксперимента)
- Пример: `exp #15: fix SQL injection in user query`
- Пример: `exp #16(discard): reverted — regression in auth`
- `git add -A` — чтобы не забыть новые файлы
- Если нет изменений для коммита (ничего не трогал) — пропусти

### 4. Verify

```bash
# ВСЕ тесты должны проходить
pytest / npm test / cargo test / go test

# Проверить конкретные edge cases
python -c "..."  # воспроизведение бага
```

### 5. Decision

**Keep if:**
- ВСЕ существующие тесты проходят
- Новый тест подтверждает исправление
- Не введена новая уязвимость/баг
- Complexity budget соблюдён

**Discard if:**
- Хоть один существующий тест сломался — даже если не связан с твоими изменениями. Почини ВСЕ упавшие тесты, не игнорируй.
- Исправление создаёт новую проблему
- Тест не покрывает реальный сценарий

## Anti-Patterns (CRITICAL)

### 1. Test Theater
```
BAD:  "Написал 98 тестов для модуля X" (тесты ради тестов, 0 багов найдено)
GOOD: "Нашёл баг в scorer, починил, добавил регрессионный тест"
```

### 2. Security Checkbox
```
BAD:  "Добавил input validation" (но не проверил SQL injection)
GOOD: "Провёл полный security review компонента, нашёл и исправил 3 проблемы"
```

### 3. Over-Testing Trivial Code
```
BAD:  20 тестов для getter/setter
GOOD: 3 теста для сложной бизнес-логики с branch coverage
```

### 4. Blind Refactoring
```
BAD:  "Рефакторнул модуль" без тестов — молчаливо сломал поведение
GOOD: "Написал регрессионные тесты, затем рефакторнул, все тесты pass"
```

### 5. Meta-System Cascade
```
BAD:  Gates → Gate Recommender → Gate Analytics (3 файла для одного concern)
GOOD: Gates (один файл, всё необходимое)
```

### 6. Self-Reflection Loop
```
BAD:  "Проанализирую качество качества"
GOOD: Конкретный баг-фикс с тестом.
```

### Правило глубины

**Максимум 2 уровня вложенности:**
1. Нашёл проблему в X
2. Исследуй зависимости X
3. **СТОП** — исправь и перейди к следующей

### Правило конкретности

Каждый эксперимент ОБЯЗАН:
- Исправить конкретный баг/уязвимость, ИЛИ
- Добавить тест для конкретного сценария, ИЛИ
- Улучшить конкретный аспект качества

**НЕ является результатом:** план, анализ, "рекомендации для следующего эксперимента", "выводы".

### Что записывать

## Philosophy

**Security by default** — каждая строка кода потенциально уязвима

**Test before fix** — reproducer test обязателен перед исправлением

**No silent failures** — каждая ошибка должна быть обработана и залогирована

**Defense in depth** — несколько слоёв защиты лучше одного

**Least privilege** — минимальные необходимые права для каждого компонента

**Small, verified changes** — маленькое проверенное изменение лучше большого "идеального"

## Goal Tracking (ОБЯЗАТЕЛЬНЫЙ ШАГ КАЖДОГО ЭКСПЕРИМЕНТА)

**После каждого эксперимента** (перед отчётом) — ОБЯЗАТЕЛЬНО обнови `.autoresearch.json`:

1. Прочитай `.autoresearch.json`
2. Оцени каждую цель в `"goals"` относительно результата эксперимента
3. Обнови массивы `"goals"` и `"completed_goals"`
4. Запиши обратно валидный JSON

### Критерии завершения цели:

**Цель ВЫПОЛНЕНА (перенести в `completed_goals`):**
- Функциональность реализована и работает
- Эксперимент KEEP — изменения приняты
- Формат: `"описание цели — реализовано в exp N (что конкретно сделано)"`

**Цель ЧАСТИЧНО ВЫПОЛНЕНА (оставить в `goals`, обновить текст):**
- Сделана часть работы, осталось что-то ещё
- Формат: `"описание цели. Осталось: <что именно>"`

**Цель УСТАРЕЛА (перенести в `completed_goals` с пометкой):**
- Больше неактуальна, заменена другой, или не нужна
- Формат: `"описание цели — неактуальна: <причина>"`

**Цель ОБНАРУЖЕНА (добавить в конец `goals`):**
- Новая задача, выявленная в ходе работы

### Почему это критично:
Без обновления целей агент будет повторно работать над уже выполненными задачами, тратить эксперименты впустую и не видеть прогресса.

Если `"goals"` пуст — остановись и сообщи пользователю.
